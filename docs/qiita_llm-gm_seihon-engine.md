# LLMにTRPGのゲームマスターをやらせると「忘れる・矛盾する」。だから状態の真実をRustの決定論エンジンに奪わせた

<!-- 公開版: https://qiita.com/outcast_zari/items/fdff41b2cd5d2d7d3506 -->

**著者: ザリ・ロブステル**

---

LLMにゲームマスター(GM)をやらせると、最初の数ターンは驚くほど面白い。問題はそのあとだ。さっき拾わせた鍵を忘れる。死んだはずのNPCが喋る。HPが文脈に応じて都合よく増減する。**LLM-GMの死因は文章力ではない。忘却と矛盾だ。**

文章力で殴り合っても勝てない領域がある。一貫性がそれだ。そこで私は逆を張った — **LLMから「状態の真実」を構造的に奪う**。HP・所持品・ダイス・フラグ・位置の唯一の真実をRustの決定論エンジンに握らせ、LLMには「提案」だけさせる。本記事はその設計と、実クラウドLLM(claude-opus-4-8)で通しプレイして**設計の前提が3つ覆った**話だ。

プロジェクト名は Kataribe(語り部)。Rustのワークスペースで、現在 63/63 テスト green。

## TL;DR

- LLM-GMの死因は忘却と矛盾。だから状態の真実をLLMに持たせない。
- **三権分立**: LLMは提案し、決定論エンジンが裁き、Memoriaが覚え、シナリオが縛る。
- LLMの出力は `StateDelta { narration, ops }`。`narration`(語り)は検証しない。`ops`(状態変更要求)は**全件検証**。検証境界をLLM出力の外、決定論コードに置く。
- **二層防御**: プロンプト層で嘘を減らし、エンジン層で弾く。
- 実機で測ったら3つの前提が覆った: ①数値閾値はLLMの自然な増分に較正が要る ②強モデルは設定文だけで伏線を自発的に語る ③意味検索は軽量なTF-IDFで足り、神経embeddingは不要だった。

## 設計: 三権分立

> LLMは提案し、エンジンが裁き、Memoriaが覚え、シナリオが縛る。

- **エンジン(正本 / Rust)**: HP/所持品/ダイス/フラグ/位置の唯一の真実。LLMはここに直接触れない。
- **LLM(提案)**: `StateDelta` を返すだけ。状態の真実は持たない。
- **シナリオ(拘束)**: `Gate` 条件つきの場所グラフ。即興が筋から外れすぎないよう縛る。
- **Memoria(記憶)**: 伏線・キャラ性格の意味検索。**可変世界状態は絶対に置かない**(曖昧な想起に可変状態を置くと「忘れるGM」を再現するため)。

肝は「正本(エンジン)」だ。LLMが何を言おうと、状態を変えるのはエンジンだけ。

## 正本エンジン: LLMから状態の真実を奪う

LLMが毎ターン返す唯一の出力形はこれだけ:

```rust
pub struct StateDelta {
    pub narration: String,   // 情景・NPC台詞。検証しない (LLMの領分)
    pub ops: Vec<StateOp>,   // 構造化された状態変更要求。エンジンが全件検証
}

#[serde(tag = "op", rename_all = "snake_case")]
pub enum StateOp {
    AddItem { item: ItemId },
    RemoveItem { item: ItemId },
    SetFlag { key: FlagKey, value: bool },
    Move { to: LocationId },
    RequestRoll { sides: u32, dc: u32 },     // 結果は含めない — エンジンが振る
    AdjustStat { entity, key, delta: i64 },  // 加減。LLMは「意図」だけ、値は持てない
    ScaleStat  { entity, key, num, den: i64 },// 乗除。den=0 はエンジンが却下
}
```

ここに設計の全体重がかかっている:

- **`narration` は検証しない。** 語りはLLMの領分だ。文章力はLLMに任せる。
- **`ops` は全件検証する。** 状態変更は構造化された要求であって、勝手な真実ではない。
- **ダイスの結果はopに書けない。** `RequestRoll { sides, dc }` には出目フィールドが無い。LLMは構造上、出目を主張できない。エンジンがseeded RNGで振り、監査可能にする。LLMが「クリティカル!」と書いても、それは語りであって判定ではない。

裁定は2つの関数に分かれる。

```rust
/// 唯一の裁定者。state を一切変更しない純粋関数。
/// 1つでも不正な op があれば Reject を返す (理由は全件収集)。
pub fn adjudicate(state: &GameState, scenario: &Scenario, delta: &StateDelta) -> Verdict;

/// adjudicate が Accept の時のみ、原子的に適用する。
/// 1つでも不正なら全体却下・state は無傷。
pub fn apply(state: &mut GameState, scenario: &Scenario, delta: &StateDelta)
    -> Result<ApplyOutcome, Verdict>;
```

2つの不変条件を守る:

1. **純粋性** — `adjudicate` は state を一切変えない。だから「却下されるべきか」を、適用せずに何度でも問える。
2. **原子性** — `apply` は、デルタ内に1つでも不正なopがあれば**全体を却下**し、stateは無傷。「鍵を開けて」+「存在しないマスターキーを取る」を束ねたら、鍵も開かない。

### 却下理由は「文字列」でなく「データ」で返す

ありがちな失敗は、却下理由を `String` で返すこと。テストが文字列一致に依存して脆くなり、多言語化もできない。

```rust
pub enum Verdict {
    Accept,
    Reject { reasons: Vec<RejectReason> },  // 構造化された理由 (12種)
}
```

`RejectReason` は `ItemNotHere { item }` / `DivideByZero { key }` のような**列挙データ**だ。文面は提示層が `localize(Lang::Ja | Lang::En)` で生成する。検証ルールはエンジンの普遍法則として固定し、文字列だけを言語層に分離する。テストは `matches!(r, RejectReason::ItemNotHere { .. })` で書けて、文言を変えても壊れない。

### 規格=実装の単一真実源

LLMにツールを使わせるとき、ツールのJSON Schemaを手書きすると、型とSchemaがすぐ乖離する。Kataribeでは `emit_delta` ツールのスキーマを `schemars` が `StateDelta` 型から**機械生成**する。手書き禁止。`StateOp` にバリアントを足すと、スキーマに自動で載る — **プロンプトの変更は要らない**。

## 二層防御: プロンプト層で減らし、エンジン層で弾く

「正本 > 文章力」を実クラウドLLMで測ったとき、面白い構造が見えた。嘘は二層で断たれる。

- **プロンプト層(嘘を減らす)**: 盤面とgate条件をLLMに提示すると、LLMは不可能な手をそもそも提案しない。密室脱出の敵対プレイで、claude-opus-4-8は「解錠前の移動」「存在しないマスターキーの取得」を**提案すらせず、語りで拒否した**。
- **エンジン層(嘘を弾く)**: それでもLLMが複数ステップを欲張って束ね、原子性違反のデルタを出すと、エンジンが全体却下し、却下理由を自然言語で返す。再生成させると、LLMは合法な部分手に修正する(2ターン目で受理)。

却下が発火したのは「欲張って束ねた」ときだけだった。**正本の原子性が「一手ずつの正しい前進」を構造的に強制する**。

ここで反証になりうる観察を一つ。Phase B で「キャラは自分の禁忌(例: 豚肉を口にしない誓い)を破れない」というエンジン強制を実装したが、強モデルではこの強制が**実機で発火しなかった**。LLMがキャラの設定文を読み、違反opを提案すらしなかったからだ。

これは失敗ではなく、二層防御の確認だ。プロンプト層(設定文=第一防衛線)が効いている証拠であり、エンジン層(禁忌=保証/backstop)の価値は別にある — **同人配布で狙う弱いローカルLLMは設定文を無視して違反を提案しうる**。そのときエンジン強制が一貫性を救う。エンジン強制の正しさは決定論テストが証明済みで、実機で見えないことは未検証を意味しない。

## 反応機構の双対性: 禁忌(却下)とトリガー(発火)は同じ機構

世界の条件は `Gate` という第一級データで表す。

```rust
#[serde(tag = "kind", rename_all = "snake_case")]
pub enum Gate {
    Always,
    HasItem { item: ItemId },
    FlagIs { key: FlagKey, value: bool },
    LocationIs { at: LocationId },
    StatAtLeast { entity, key, value: i64 },  // 数値条件
    All { of: Vec<Gate> },
    Any { of: Vec<Gate> },
}
```

ここで気づいたことがある。**「delta適用後にGateが真化したか」を判定する単一機構の上に、二つの双対操作が乗る。**

- **禁忌(負)**: 真化する遷移を**却下**する。「豚肉を食べた」フラグが false→true になるデルタを拒否。
- **トリガー(正)**: 真化したら**発火**する。「好感度 ≥ 30」になった瞬間、効果を適用し、語りを注入する。

実装はこうだ。`adjudicate` は純粋なので、delta を state の clone に射影(project)して評価する:

```rust
// 禁忌: clone へ射影し、taboo(Gate) が false→true に真化するなら却下
fn check_taboos(reasons, state, scenario, delta) {
    let mut projected = state.clone();
    apply_ops(&mut projected, scenario, delta);
    for taboo in /* 各キャラの禁忌 */ {
        if !taboo.eval(state) && taboo.eval(&projected) {
            reasons.push(RejectReason::TabooViolated { .. });
        }
    }
}
```

トリガーはこの鏡像だ。受理・適用後の実stateで、`when` が真でまだ発火していないものを発火させ、authored な効果を適用し、発火済み集合(`fired`)にラッチする。効果が次のトリガーを真化させる連鎖は、新たな発火が無くなるまでsettleさせる(各トリガーは高々1回発火するので必ず停止)。

**同じ「post-transition述語評価」機構を、却下と発火で共有する。** 禁忌を先に実装すれば、トリガーはその機構の再利用で済む。世界の一貫性(忘れない)と、物語の反応性(伏線が必ず回収される)を、両方エンジンが保証できる。

## Memoria: 可変状態は禁忌、伏線だけ覚える

「思い出す」系のトリガーは、記憶層(Memoria)への自然な橋渡しになる。条件成立 → 伏線を想起 → 語りに注入。

ここで北極星の不変条件がある。**Memoriaには可変世界状態を絶対に置かない。** 持てるのは不変の伏線・キャラ性格(authored lore)だけ。これを型で構造的に保証した:

```rust
pub struct MemoryFragment {
    pub id: String,        // 想起の主キー
    pub tags: Vec<String>, // 別名キー
    pub text: String,      // 語りに注入する伏線の本文
}
// HP も所持品もフラグも、フィールドとして存在しない。
// recall(&self) は &self — 世界状態を変えられないことの型レベル保証。
```

想起は「決定論ベクトル」でやる — **文字bigramのTF-IDF cosine類似**だ。日本語は単語境界が無いので文字n-gramが頑健で、依存ゼロ・ネット不要・テスト可能。authored cueの完全一致(id/tag)はスコア1.0で常に最上位にし(旧挙動の上位互換)、その上に意味的に近い断片をcosineランクで足す。

神経embedding(候補: candle + 小型sentence-transformer)を引き込む案もあったが、後述の実測で**不要と判明した**。

## 実機で測ったら設計の前提が3つ覆った

ここからが本題だ。実クラウドLLM(claude-opus-4-8 @ Anthropic互換)で通しプレイして、机上では見えなかった前提が3つ覆った。

### 発見1: 数値閾値はLLMの自然な増分に較正せよ

「好感度が30に達したらトリガー発火」というシナリオを実機で回した。プレイヤーがアリスと打ち解ける行動を入力し、LLMが好感度を上げる。結果:

```
turn1 好感度=1   turn2 好感度=3   turn3 好感度=5   turn4 好感度=8
```

**LLMは好感度を +1〜3/ターンと、realisticに小刻みに動かした。** 4ターンで8止まり。閾値30には全く届かず、トリガーは発火しなかった。

決定論テストでは `raise_affection(30)` を1デルタで直接入れていたので、この乖離が見えなかった。**test-authoredな値はエンジンの正しさを証明するが、運用パラメータの妥当性は証明しない。** 閾値を実機の増分相場(好感度なら1イベント+1〜3)に合わせて初めて、発火・連鎖・ゴール到達を観察できた。

### 発見2: 強モデルは設定文だけで伏線を自発的に語る

発火前のターンで、すでにLLMはこう語っていた:

> 「子どもの頃ね、雪の日に……この暖炉の前で、誰かと何かを約束した気がするのよ。誰だったかは、どうしても思い出せないんだけど」

トリガーも記憶注入も発火していないのに、だ。LLMがキャラの設定文(「幼い頃に主人公と約束を交わした——本人はまだ思い出していない」)を読み、伏線を**自分で先取りした**。

含意は重い。**「LLMに思い出させること」自体は、強モデルでは設定文だけで足りる。** では記憶層(memoria_bridge)の固有価値は何か。surfacing そのものではなく、**①精密な閾値での発火保証(決定論) ②常時可視の設定文に載せきれない大規模loreのon-demand想起**、の2つに絞られる。小さな設定文で済む盤面では、記憶層を無理に使わない判断もありうる。

### 発見3: 意味検索はTF-IDFで足り、神経embeddingは不要だった

閾値を実機向けに下げた盤面で、想起のcueをわざと曖昧な文字列にして回した:

```yaml
recall: 丘の樫の木で交わした幼い約束   # 伏線のidでもtagでもない曖昧文字列
```

実機出力(✦がトリガー発火、┊が想起された伏線):

```
  ✦ アリスの表情がふと遠くなり、長く忘れていた約束の記憶がよみがえる。
    ┊ 幼い二人は、丘の上の古い樫の木の下で、小指を絡めて誓った——
    ┊ 「離れても、いつか必ずここへ戻る」と。
  ✦ 思い出した約束を、アリスは今あらためて心に誓い直す。

🎉 クリア。goal 到達 (turn 3).
```

cueがidともtagとも一致しないので、TF-IDF cosine経路が発火し、正しく伏線を想起した。連鎖発火(想起→誓い直し)とゴール到達まで端から端まで通った。

ここで反証点を明示する。**TF-IDFが効いたのは、cueと伏線本文が語彙(丘/樫の木/約束/幼)を共有していたからだ。** 語彙が完全に乖離したparaphrase(例: cueに「樫」も「約束」も無いのに意味は同じ)を跨ぐなら、文字n-gramでは届かず、神経embeddingが要る。

だが — **cueをシナリオ作者が書く現設計では、その状況は発生しない。** 作者はidかtagで完全一致させるか、本文と語彙の重なる曖昧cueを書けばいい。「完全一致=保証」+「語彙の重なる曖昧cue=cosineで届く」の二段で実用上十分だ。

結論: **神経embeddingは引き込まない。** 100MB超のモデル重みとビルド複雑化を、推測でなく実測で回避できた。必要になってから足せばいい(同じtraitの裏で差し替え可能に設計してある)。

## 罠台帳から: 互換APIの現実

実装中の罠は全部 `failures.md` に1件1エントリで残している(現在22件)。LLM接続まわりで効いたものを抜粋:

- **新しめのモデルは `temperature` を送ると400で弾く。** claude-opus-4-8は `temperature is deprecated for this model` を返す。教訓: 互換APIでも「全provider共通の必須パラメータ」は思ったより少ない。**送る前提でなく、省略を既定にしてprovider既定に委ねる**のが最も壊れにくい。tool_choice強制が構造を保証するので、温度固定はそもそも不要だった。

- **parallel tool useは既定ON。** モデルはツール呼び出しを複数返しうる。`tool_calls.first()` だけ読むと残りを黙殺する — 「矛盾しない」に反する潜在バグだ。単一ツールをtool_choice強制している現状では未発火だが、複数検出時は明示エラーかログ化すべき。

- **却下→再生成で、tool_use→tool_resultプロトコルを意図的に回避した。** 正規には forced tool の応答後、対応する `tool_result` を返すのが筋だ。だがKataribeの再生成は、LLMの提案をプレーンなassistantテキストでechoし、却下理由をuserテキストで積む。履歴にdangling tool_callを残さないので tool_result 要求を回避できる。我々は「ツール出力に反応させる」のでなく「再提案させる」ためだ。この設計判断は実Anthropic APIで通ることを敵対プレイで実証した。

- **`from_env()` が `dotenv()` を呼ぶとテスト不能になる。** .env読み込みはアプリ入口(bin)の責務に移す。ライブラリは環境を勝手に書き換えない。テスト不能は設計の臭いだ。

## まとめ: 検証境界をLLMの外に置く

Kataribeの設計を一行にすると、**検証境界をLLM出力の外(決定論コード)に置く**だ。

- LLMには構造化出力(tool-use強制)を出させ、その`ops`を決定論エンジンが全件検証する。
- 却下理由は機械可読なデータでなく、自然言語でLLMに戻す(再生成の燃料にする)。
- ツールスキーマは型から機械生成し、規格と実装の乖離を断つ。
- 数値・ダイス・状態遷移はエンジンが代行し、LLMは「意図」だけ提案する。

この型は、TRPG-GMに限らない。**「LLMが提案し、決定論層が裁く」任意のシステム** — エージェント、ツール実行、状態機械駆動 — に転用できる。

そして実機で測って初めて分かったことが3つあった。閾値はLLMの自然な行動で較正する。強モデルでは決定論層の価値は「surfacing」でなく「保証とスケール」にある。意味検索はauthored cue設計なら軽量手法で足りる。**机上の設計は仮説にすぎず、接地は実測でしか得られない。**
