# spec 18: プレイヤーが振るダイス — 開帳・選択・交互

**Status**: rev2 / Phase A 実装済み（2026-07-20 起草 → 同日査読 7+CoC7 準拠表反映 → 同日 Phase A。B/C は Draft、B は spec 16 実測後）
**依存**: spec 16（d100 ロールアンダー・成功度）/ spec 09（逐次射影）/ spec 14（入力キャッシュ）

## 動機（ユーザー、2026-07-20）

> TRPG は基本プレイヤーがダイスを振るから緊張感がある。でも Kataribe は正本システムが
> 振ってしまうので、TRPG をやっている感覚がなく緊張感もない。ダイスはプレイヤーが
> クリックで振るようにしたい（ソシャゲのガチャのようなドーパミン）。
> 戦闘などは、プレイヤーと正本が LLM を介さず交互に振り、結果だけ LLM に伝えたい
> （トークン消費を減らせる）。

## 前提の接地

- **出目はいつ振っても同じ**: ダイスは seeded RNG（`RngState`）で、seed と消費順から決まる。
  クリックの瞬間に engine が振るよう改造しても、**出る数字は変わらない**。「クリックが結果を
  決める」は原理的に幻。
  - **正確な決定論の定義（rev2 査読 I-1）**: 保証は「同 seed + **同じ選択列** → 同じ出目」。
    プッシュするか・Contest を何回すかで消費順は変わるが、これは現行 engine でも
    「いつ challenge に挑むか」で変わるのと同じ。`RngState` はストリーム位置ごとセーブ
    されるので追加の台帳（消費カウンタ等）は不要。Phase A（開帳のみ・分岐なし）では
    文字どおり「いつ開いても同じ」。
- **ガチャも同型**: ソシャゲの排出はサーバ側で確定済みで、演出は**開帳**である。緊張の源は
  決定の遅延ではなく「結果を知らない時間」と「引くという自分の行為」。
  → **提示層だけで狙いの体験は出せる**（engine を触らずに済む、が本 spec の第一の梃子）。
- **出目を本当に変えるのは選択だけ**: どの技能で挑むか、押すか退くか、幸運をいくら払うか。
  CoC7 にはこれが最初から在る（プッシュロール・幸運消費）。**緊張の本体は②**。
- 現状の経路: 判定は `apply` の中で解決し、`CheckOutcome`（+ authored 結末文・SE）が
  `TurnView.checks` に載って受理と同時にログへ出る。`rolls`(RollView) / `stat_rolls`(spec 16) も同様。

## 北極星整合

- **engine は振る主体のまま**（LLM も UI も出目を持てない）。変わるのは*見せ方*と*選択肢*で
  あって、誰が数字を作るかではない。
- **プレイヤーの選択も engine が裁く**（幸運が足りなければ却下、押せない判定は押せない）。
  第三の入力経路が増えても「正本 > 文章力」は不変。
- **決着はゴール/フラグに接地**（2026-06-27 の「山場を 1 判定で潰させない」）。③ の交互振りは
  authored な交換回数で山場を割る装置であり、engine が戦闘の勝敗を語り始める話ではない。

## 設計

### ① 開帳（Phase A・提示層のみ・engine 無改修）

受理ターンに `checks` / `rolls` / `stat_rolls` が在れば、**結果を伏せてダイスを出す**。
クリックで開く。

- 伏せる対象は**判定行そのもの**と、それに付随する **authored 結末文**・**結末 SE**
  （結末文が先に見えたら開帳の意味がない。SE は開帳と同時に鳴らす）。
- **1 クリック 1 判定**（同一ターンに複数在れば順に開く）。束ねて一度に開くのは、
  複数判定が並ぶ盤面（対抗・連続判定）で緊張が潰れる。
- GM の語り（narration）は先に出す。判定結果に言及しない設計（結果は次ターン還流か
  authored 結末文）なので、**語り → ダイス → 結末** の順が自然に成立する。
- **未開帳のまま次のターンへ進めない**（入力欄を締める）。積み残しは体験を壊す。
- 設定に**オフ**（即時開示＝従来動作）を置く。演出が邪魔な作者テストのため。
- 会話ログのテキスト保存は開帳済みの体裁で書く（伏字を保存しない）。
- **リロードは自動開帳（rev2 査読 I-4）**: 未開帳は frontend の揮発状態なので、アプリを
  閉じて再開すると開いた状態で戻る。「必ずクリックさせる」は不変式ではなく演出 —
  リロードという迂回路は**許容**と明記する（Phase B の決断は GameState 持ちなので
  こちらはセーブを跨いで残る。演出と決断で永続性が違うのは意図）。

**engine・prompt・セーブは無改修**。

### ② 選択（Phase B・CoC7 の本丸）

CoC7 の二つの決断を op にする。**これが「クリックが演出でなく決断になる」段**。

- **プッシュロール**: 失敗した判定を、代償を払って 1 度だけ振り直す。
- **差分の買い取り**: 失敗した出目と目標値の差分を、**どこかの stat から支払って**成功に変える。

**支払い元は authored（2026-07-20 ユーザー決定・本 spec の設計の要）**。
engine に `LUCK` という固定変数は作らない — **どの stat から引くかをシナリオが書く**。
stat キーは既に開集合（宣言されたものだけが存在する閉世界）なので、`幸運` でも `所持金` でも
`霊力` でも engine は名前を解釈せず宣言に従う。CoC7 は「幸運」を宣言した一例にすぎない、
という位置づけにする（原典に engine を寄せない）。

```yaml
# シナリオ全体の既定 (challenge 単位で上書き可)
spend_rules:
  from: 幸運        # 支払い元 stat (作者の自由なキー。宣言済み必須)
  rate: 1           # 1 ポイントで差分 1 を埋める
push_cost:
  from: hp          # 押す代償 (多くは hp、金でも正気度でもよい)
  amount: 1
```

- **払える上限は engine が出す**: `現在値 − min 宣言`（`StatDecl.min`）。足りなければ買い取りを
  提示しない／提案されても却下する（`InsufficientStat` 相当の却下理由）。**払って死ぬ**設計も
  作者が `min: 0` と goal（`stat_at_most hp 0`）で書けば成立する — engine は止めない。
- **未宣言キーは load 時に弾く**（`ChallengeStatUndeclared` と同型の validate）。幻の財布を作らせない。
- **提示**: 失敗が開帳された瞬間に「あと 7 足りない。幸運を 7 払えば成功にできる（残 48）」と
  出し、クリックで払う。**払わない自由が緊張の本体**なので、自動で引くことはしない。
- 加算式（ロールハイ）でも同じ式で成立する（差分を合計に足す）。

設計論点（本 spec の核心・Phase B 着手時に決める）:

- **「プレイヤー op」という第三の権能**: いまの op は *LLM 提案* か *authored 専権* の
  二種しかない。押す/払うは**プレイヤーが UI から直接 engine に入れる**op で、三権分立に
  新しい辺が増える。LLM を介さない（＝トークンを使わない）のが利点で、③ とも地続き。
  turn loop との関係（rev2 査読 I-7）: プレイヤー op は **GM ターンの間**（受理表示後・
  次入力前）に同期で解決し、結果は `pending_checks` / `check_outcome_note` に積んで
  **次の GM ターン**が読む — 既存の「判定結果は次ターン還流」の輪に乗るだけで、
  `TurnView` の生成順序は変えない。
- **帰結の確定遅延（rev2 で判明した最重量の設計判断・査読の指摘の一段先）**: 現行の
  challenge は apply の中で on_failure のフラグ・effects まで**原子適用済み**。プッシュで
  振り直しても、適用済みの帰結は巻き戻せない（任意 op の逆操作は定義できない）。
  → **pushable な challenge は帰結の適用を決断確定まで遅延する**: apply は出目と degree を
  計算して `pending_decision` に凍結し、フラグ・effects・トリガー発火は**決断
  （押す/払う/受け入れる）が確定した時点**で原子適用する。非 pushable な challenge は
  従来どおり即時適用（挙動不変）。この遅延が spec 09 の逐次射影・trigger settle と
  どう噛むかが Phase B の PoC の中心。
- **開帳と決断の直列化（rev2 査読 I-2/I-3）**: 「一つしか持てない pending_decision」と
  「1 ターン複数判定」の衝突は、**開帳 → 決断 → 次の開帳**の直列フローで解消する。
  判定は一つずつ開き、pushable なら決断（押す/払う/受け入れる）を終えてから次を開く。
  よって同時に存在する未決断は常に高々一つ = 単一スロットで足りる。
  **全部開き終わり・全決断が済むまで入力欄は締める**（「開いたが未決断」も締め対象）。
- **`PendingDecision` の素性（rev2 査読 I-5）**: `{ challenge_id, entity, stat_key,
  raw_roll, target, degree, pushed: bool }`。**raw_roll 必須** — 差分買いは
  `raw_roll − 買いたい成功度の閾値` の計算に生の出目が要る（degree だけでは出せない）。
  セーブ対象（spec 07 の器・serde default で旧セーブ互換）。
- **押す代償は既定なし（rev2 査読 I-6・原典準拠）**: CoC7 のプッシュは stat コスト無しで
  「失敗したらより悪い帰結」が代償。よって `push_cost` は **Option（既定 None）**、
  代償の本線は `on_push_failure`（無ければ `on_fumble` → `on_failure` のフォールバック連鎖、
  spec 16 の `resolve_degree_slot` と同流儀）。`push_cost` は「押すのに hp/金を払う」を
  作者が**足したい時だけ**書く上乗せ（二重取りは作者の意図した時だけ起きる）。
- **結果の還流**: 押した/払ったことは GM が知る必要がある（語りが噛み合う）。
  `check_outcome_note` に「押して振り直した（1 度目: 失敗 62 → 2 度目: 成功 31）」
  「幸運を 7 払ってハード成功に変えた（残 41）」を載せる。

**押せる/買える の規律（rev2 査読 II を構造と authored の二層で取る）**:

engine は stat の名前（幸運/SAN/…）を解釈しない — 閉世界の根幹。CoC7 の禁止リストは
**構造**と **authored 宣言**で同等物を取る:

- **構造で不可能（op が存在しない）**: ダメージ・SAN 減少は `RollStat`＝authored 専権
  なのでプレイヤーが押す入口が無い。対抗（③ Contest）は**既定 pushable: false**。
- **engine 既定（様式に依らず健全なので engine が持つ）**:
  - プッシュは**失敗した判定のみ**・**1 度だけ**（`pushed` フラグ）。
  - **ファンブルは買えない**（degree==fumble なら spend を提示も受理もしない）。
  - **プッシュと差分買いは排他**（押した判定は買えない、買った判定は押せない —
    CoC7 準拠であると同時に、無限リソース変換の穴を塞ぐ engine 健全性）。
  - プッシュ後の再ファンブルは買えない（同上）。
- **authored 宣言**: `ChallengeDef.pushable: Option<bool>`（既定 true）と
  `ChallengeDef.spendable: Option<bool>`（既定 true）。幸運ロール自体・重要な一発勝負を
  作者が締める。`Scenario.spend_rules` が無ければ差分買いはそもそも出ない（opt-in）。
- **成功度の買い上げ（rev2 査読 II）**: 差分買いは regular 止まりでなく **hard/extreme
  まで買える**（critical=01 は出目そのものなので買えない）。UI は「あと 7 で成功、
  あと 19 でハード成功（残 48）」と**段階ごとに提示**し、どこまで払うかも選択にする
  （`spend_rules.rate` はそのまま各段に適用）。

### ③ 交互振り（Phase C・トークン経済）

「このキャラとこのキャラを戦わせる」を、LLM を挟まず engine とプレイヤーで交互に解決し、
**digest だけを LLM に渡す**。1 ターン＝フルプロンプト 1 往復なので、3 交換を 3 ターンに
割ると入力が約 3 倍かかる（spec 09 の束ね受理と同じ経済）。

- **最小プリミティブ案 `Contest`**: 対抗ロール 1 回（双方 authored 主体・双方の stat で振り、
  degree または合計で優劣を決める）。「戦闘」という概念は engine に入れない。
  交換の回数・打ち切り条件・ダメージは authored（challenge の帰結・トリガー）で書く。
- **キャラ別のダイステンプレート（2026-07-20 ユーザー決定）**: 「このキャラが振るときはこれ」を
  **キャラクターファイルまたはシナリオに書く**。`Contest` はテンプレート名で相手を指すだけ
  になり、素性（stat・面・修正・様式）は常に authored。**LLM は選ぶだけで数字を持てない**
  という `AttemptChallenge` の思想（閉世界）を対抗ロールへ延長したもの。

  ```yaml
  # characters/ogre.yaml — そのキャラ自身の振り方
  rolls:
    噛みつき: { stat: 格闘, sides: 100 }        # percentile (spec 16)
    回避:     { stat: 敏捷, sides: 20, bonus: 2 } # 加算式
  ```

  - **解決順は inline 優先**（`Scenario.rolls[entity][id]` が外部 `CharacterDef.rolls` を上書き
    ＝ `inject_cast` と同じ流儀）。同じ ogre を別シナリオで別の振り方にできる。
  - **未宣言テンプレート・未宣言 stat は load 時に弾く**（プレイ中の `UnknownStat` より早く、
    どのキャラのどのテンプレートかを名指しで）。
  - 様式（ロールアンダー / 加算式）は**テンプレート単位**で決まる。`check_style` の既定に
    従いつつ、テンプレートが明示すればそれが勝つ（percentile の敵と加算式の味方が同席できる）。
  - `Contest` の解決は degree（percentile）どうし、合計（加算式）どうしの比較に閉じる。
    **様式を跨いだ対抗は禁止**（validate で弾く — 比較の意味が定義できない）。
- **UI**: プレイヤー側の振りは ① の開帳と同じクリック。engine 側（相手）の振りは即開示。
  交互に数回、その間 LLM は呼ばない。
- **digest**: `3 交換: 命中/回避成功/致命 → hp 12→5` 相当の 1 行を組み、次の GM ターンに
  「直前の判定結果」として還流する（既存の `check_outcome_note` の拡張）。
- **境界の見極めが Phase C の主題**: engine が持つのは「対抗ロール 1 回」まで。
  イニシアチブ・射程・武器表を持ち始めたら北極星から外れる。

## 実装 Phase

- **✅Phase A（開帳、2026-07-20 実装）**: frontend のみ（engine/prompt/セーブ無改修）。
  - ダイス系 3 行（rolls/checks/stat_rolls）に `revealed` カウンタ。伏せカード
    `DiceReveal.vue` = クリック → 数字スクランブル（power2.out 風減速 1.1s、サンプルの
    デコード演出を CSS+JS 移植・依存ゼロ）→ **確定出目に着地** → 本物の行に差し替え
    （着地と結果の数字が同じ = 連続性）。
  - **1 クリック 1 判定の直列**: 開けるのは常に最初の未開帳 entry（`revealTargetIndex`）。
  - **漏洩防止が実装の過半**: ダイスより後ろの行（ビート/goal バナー/遷移/エピローグ）は
    `pendingTail` に保留し全開帳で flush。**ビート SE・イベント CG 背景・BGM も保留**
    （発火 CG は判定の帰結でありうる = 開帳前に見えたら漏洩）。結末 SE は開帳と同時。
    既知の残漏洩: 右ペイン状態パネル（hp 等）は即時更新 — Phase A では許容し実プレイで
    気になるか測る。ログのテキスト保存も保留中の tail は含まない（開帳後に保存すれば全量）。
  - 未開帳中は入力欄を締める（`hasUnrevealedDice`）。設定「グラフィック」にオン/オフ
    （`kataribe.diceReveal`、既定 on）。**オフ切替は revealAll = 入力ロックの脱出口**。
  - 新規/再開/ロードで保留は必ず破棄（揮発）。**実プレイで手触りを測るのが本体**（未知 1・2）。
  - **✅実機目視 (2026-07-20, dice_trial + 実 LLM)**: 通しプレイで trial_cleared 到達を確認
    （breakthrough CG 差し替え・goal バナー・エピローグまで動線一式）。GM が world の
    「運命は既に転がっており、挑む者はそれを開くだけ」を語りに自発的に織り込む副産物も。
    未知 1（緊張）・2（テンポ）の定量は継続プレイで。
- **Phase B（選択）**: `GameState` の押下可能判定 + プレイヤー op 2 種 + authored 代償 +
  還流。gm_core に触る（PoC 必須）。**spec 16 の実 LLM 実測を先に片付けてから**着手する
  （percentile の土台が未実測のまま上物を積まない）。
- **Phase C（交互）**: `Contest` プリミティブ + digest 還流 + UI。着手前にスコープの線を
  data_contract に凍結する。

### CoC7 準拠表（rev2 — 何で取るか / 取らないか）

| CoC7 要件 | spec 18 の取り方 |
| :--- | :--- |
| d100 ロールアンダー + 成功度 5 段階 | ✅ spec 16 実装済み（依存） |
| プッシュ = 失敗時のみ・1 度だけ | engine 既定 |
| プッシュ失敗はより悪い帰結 | `on_push_failure`（authored、フォールバック連鎖） |
| プッシュ禁止（幸運/SAN/戦闘/ダメージ/対抗） | 構造（RollStat は op 不在・Contest 既定 false）+ `pushable: false`（authored） |
| 幸運消費 = 差分 1:1 買い取り | `spend_rules`（authored opt-in・支払い元は自由な stat） |
| hard/extreme まで買える | engine（段階提示）。critical は買えない |
| ファンブルは買えない | engine 既定 |
| プッシュ ↔ 幸運消費の排他 | engine 既定 |
| ボーナス/ペナルティダイス | **❌ 非準拠のまま**（決定事項 5 — 分布の異なる目標値加算近似。準拠を謳わない） |
| 幸運の回復（セッション毎 1d10 等） | engine は持たない。authored トリガー（`adjust_stat`/`roll_stat`）で書ける |

**本 spec は「CoC7 完全準拠」を謳わない**。緊張の本体（押すか退くか・払うか払わないか・
禁止と排他）は engine 既定+authored で原典同等を取り、ボーナスダイスと幸運回復は
据え置き/authored に倒す。

## 核心的未知（実プレイで測る）

1. **開帳だけで緊張が出るか**（①の賭け）。出なければ②へ急ぐ根拠になり、出れば①だけで
   体験の大半が取れたことになる。
   **初回計測 (2026-07-20, ユーザーFB)**: 「すぐに決まるよりクリックがあったほうが
   **決断しているという雰囲気**になる」— 陽性。開帳（実際は決定ではない）だけで主体感が
   出る = 演出が主体感を作れるなら、Phase B の押す/払う（本物の決断）はその感覚に実体を
   与える増築になる。n=1 の定性、継続プレイで追認。
2. **未開帳ブロックの鬱陶しさ**: 「振る」を挟むことがテンポを削がないか。連続判定の盤面で測る。
3. **プレイヤー op が turn loop に馴染むか**: LLM を介さない state 変更が、GM の語りの
   連続性（#27 系）を壊さないか。
3.5. **帰結の確定遅延が既存機構と噛むか**（rev2）: pushable challenge の遅延適用が
   spec 09 の逐次射影・trigger settle・同ターンの後続 op とどう相互作用するか。
   Phase B の PoC はここから書く（Red が最初に出る場所）。
4. **digest で LLM が戦闘を語れるか**（③）: 1 行の要約から、交換の手触りを再構成できるか。

## 決定事項

1. **順序 = 開帳 → 選択 → 交互**（2026-07-20 ユーザー決定）。engine に触るのを後回しにでき、
   ①の実測が②③の必要性を較正する。
2. **クリックは開帳であって決定ではない**と明示的に受け入れる（seeded RNG の性質。
   「クリックした瞬間に振る」ようengine を改造しても出目は同じなので、その改造はしない）。
3. **支払い元・押す代償は authored な stat 参照**（2026-07-20）。engine に `LUCK` のような
   固定変数を作らない。CoC7 の「幸運」は作者がそう宣言した一例で、金・正気度・霊力でもよい。
4. **振り方はキャラ別テンプレートとして authored**（2026-07-20）。キャラクターファイル
   （または inline のシナリオ）が「このキャラはこう振る」を持ち、`Contest` は名前で指す。
5. **CoC7 完全準拠は謳わない**（rev2 査読 II の裁定）: 押す/買うの禁止・排他・段階買いは
   engine 既定+authored で原典同等を取るが、**ボーナス/ペナルティダイスは目標値加算の近似の
   まま**（spec 16 の明文化を維持）。原典の分布が要る需要が実測されたら `ChallengeMod` に
   `bonus_dice` を足す別 spec（10 の位を複数振る機構は RollView/監査表示まで波及するため）。
6. **プッシュ/差分買いの禁止は engine のカテゴリ名でなく構造+宣言で取る**（rev2）:
   engine は stat の名前を解釈しない（閉世界の根幹）。ダメージ/SAN は RollStat＝op 不在、
   対抗は既定 false、残りは `pushable`/`spendable` の authored 宣言。
7. **RNG 消費カウンタは新設しない**（rev2 査読 I-1 への裁定）: `RngState` がストリーム位置
   ごとセーブされるので決定論は既に器がある。「同 seed + 同じ選択列 → 同じ出目」を
   ドキュメントとして明文化するに留める。

## スコープ外（据え置き）

- ダイスの 3D 描画・物理演出（CSS のごく短いアニメーションに留める）。
- ボーナス/ペナルティダイス（CoC7 原典の分布）— spec 16 で「modifiers は目標値加算の近似」と
  明文化済み。必要になったら `ChallengeMod` 側に dice を足す。
- イニシアチブ・射程・武器表・行動順の管理（engine に戦闘システムを持たせない）。
